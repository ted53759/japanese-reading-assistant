# Project FAQ

這份文件整理 Japanese Reading Assistant 的常見設計問題，包含專案定位、與一般 AI 翻譯的差異、主要技術挑戰、prompt 與輸出驗證設計，以及後續改進方向。

---

## 這個專案解決什麼問題？

小說、故事和角色對話很適合長期學習日文，因為情境比教科書例句豐富，也更容易讓人想繼續讀下去。但對初級或中級學習者來說，直接讀原文小說門檻很高：閱讀過程會被頻繁的查單字打斷，而且就算查到單字，也不一定能理解句子裡助詞跟句尾語氣帶出的完整結構。久了之後，原本有趣的小說反而變成負擔。

Japanese Reading Assistant 想解決的就是這個問題。它會把日文長文本（EPUB 或 TXT）轉換成「原句、逐字拆解、假名、中文意思、自然翻譯」的格式，讓學習者不用一直中斷閱讀去查字典，也能看懂句子的組成，用有趣的原文材料持續學習。

---

## 這跟直接叫 AI 翻譯有什麼不同？

如果只看單一句子，直接把句子丟給 AI 翻譯確實可以得到類似的結果。這個專案的重點不在「翻得比較準」，而在如何處理整本小說、幾萬字的長文本，而不是單一句子。

如果只是把整篇文字直接丟給模型，會馬上遇到幾個問題：輸入超過 token 限制、輸出格式不穩定導致無法自動合併、中途失敗或某段輸出品質不好時很難定位問題，也很難只重跑有問題的那一段。

因此這個專案在「呼叫 AI」之外，做了一整層工程處理：把長文本切成不破壞語意的句子與 chunk、讓每個 API 輸入輸出都可以被追蹤、驗證輸出格式是否符合規範、在失敗時只重試有問題的部分而不是整批重跑。AI 呼叫只是流程中的一個步驟，前後的切分與驗證才是這個專案實際花心力解決的部分。

---

## 最大的技術挑戰是什麼？

讓 LLM 輸出格式保持穩定，比想像中困難。

即使 prompt 已經寫得很明確，模型還是會偶爾漏掉某個編號、把整句原文直接複製進逐字拆解區、中文解釋裡混入其他語言的字元，或是把助詞黏在前一個詞後面沒有拆開。這些問題如果沒被抓出來，會直接影響最終教材的品質；而且因為是長文本批次處理，一旦某個環節出錯又沒被發現，錯誤會一路帶進最終合併的檔案。

解法是把「生成」與「驗證」分開處理：先用明確的規則定義 prompt，再在 API 回傳後另外寫一層檢查邏輯，針對編號完整性、格式正確性、語言混雜等情況做檢查，發現問題就自動重試，重試仍失敗就先隔離保存，不讓它進入最終檔案。這也是這個專案的核心心得：AI 應用如果完全信任模型輸出，是沒辦法穩定運作的，需要獨立的驗證層。

## Prompt 與輸出驗證是怎麼設計的？

**Prompt 設計**

System prompt（詳見 [prompts/word_breakdown_prompt.md](../prompts/word_breakdown_prompt.md)）定義了嚴格的輸出格式規則，包括：

- 每個【編號】後面必須保留完整日文原句，不可省略任何編號或句子
- 逐字拆解區每個詞語獨立一行，格式固定為「單詞 （假名 / 中文）」
- 助詞（は、が、を、に…）必須獨立拆出，不能黏在前一個詞後面
- 句尾語氣詞（んです、よね、でしょう 等）不可省略
- 中文解釋只能使用繁體中文、必要英文或數字，不可混入其他語言字元
- 標點符號不拆解，短反應句（はあ、え、うん 等）仍需拆解

**輸出驗證**

API 回傳結果後，程式會再檢查一次格式，項目包含：

- 是否漏掉句子編號、是否出現重複或範圍外編號
- 是否缺少日文原句、逐字拆解或中文翻譯
- 是否把整句原文誤放進逐字拆解區
- 是否出現括號格式錯誤

發現嚴重問題時會自動重試；重試後仍失敗的輸出會另外保存，不會進入最終合併的檔案，避免錯誤內容混入教材。

## 後續可以改進的方向

- 提供簡單的 Web UI，讓使用者不需要透過指令稿操作
- 讓使用者可以自訂每個 chunk 的句數，不固定為 8 句
- 支援匯出 PDF，方便離線閱讀
- 支援更多輸出格式，不僅限於 Markdown
- 加入詞彙複習功能，例如將逐字拆解結果轉成 Anki 卡片，讓查過的單字可以被系統性複習
